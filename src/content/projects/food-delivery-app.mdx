## ğŸ§  Problem Statement

In some rural areas in the Philippines, the use of food delivery platforms like
**Food Panda** and **Grab Food** remains underdeveloped, despite the growing
presence of local restaurant businesses within these communities. Recognizing
this gap, our team set out to build an effective and accessible food delivery
solution tailored for these underserved areas.

As the sole backend developer on the team, I was responsible for building the
entire server-side infrastructure. The current deployment serves as a
preliminary production release that demonstrates my contribution and provides a
foundation for the rest of the team.

The backend was developed using **GoLang**, **Gin framework**, and **GORM ORM**
which provides a modern, reliable and scalable API to connect **Customers**,
**Drivers**, and **Shop Owners**. It follows **clean architecture principles**
to ensure separation of concerns and long-term maintainability.

With built-in support for **real-time order updates using WebSockets**, the
system ensures seamless communication across all roles. While some members of
the team are still in the early stages of learning frontend development, I
continue to provide guidance as we work toward completing the full application
collaboratively.

clear separation of concerns and long-term maintainability. The project is
organized into the following key layers:

### ğŸ§© Feature-Based Internal Modules

Each feature (e.g. `auth`, `order`, `restaurant`, `user`, `notifications`) lives
inside the `internal/` directory, and contains:

- `handler.go` â€“ defines HTTP handlers for the Gin router
- `service.go` â€“ implements business logic and use cases
- `repo.go` â€“ interacts with the database via GORM
- `dto.go` â€“ defines data transfer objects for request and response schemas

This structure makes each feature independently testable, easy to maintain, and

## ğŸš€ Key Features

ğŸ” **Custom Authentication Flow**

- **Customers** sign up via **OAuth** (e.g. Google) and must pass **SMS-based
  OTP verification** before gaining access
- **Drivers** and **Shop Owners** cannot self-register wherein they are created
  and managed by an **Admin** via protected endpoints

ğŸ›¡ï¸ **Role-Based Access Control (RBAC)**

- Routes are protected and filtered depending on the authenticated user's role.
- Each role (Customer, Driver, Owner, Admin) has access only to the appropriate
  subset of API functionality

ğŸ” **Full Food Delivery Transaction API**

- **Customers**: browse restaurants, place orders, track order status
- **Shop Owners**: manage their restaurants, update menu items, accept/reject
  orders
- **Drivers**: accept delivery assignments, update delivery status
- **Admin**: onboard new owners and drivers, monitor platform activity

ğŸ“¡ **Real-Time Order Tracking with WebSockets**

- Instant updates between Customer, Owner, and Driver as the order progresses
  from creation to delivery
- WebSocket endpoints designed for role-specific listeners (e.g. live updates
  for incoming orders or delivery status changes) x
